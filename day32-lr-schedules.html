📈 Learning-Rate Schedules — the LR is the single most important knob, and changing it over training — a warmup ramp, then a cosine glide down to zero — often decides convergence more than the number you pick

Same model, same data — only the shape of the learning rate changes. Watch it decide whether the optimizer converges, crawls, or blows up.

A schedule is just a function lr(t) that returns the step size at training step t. Pick a schedule and its parameters on the left and read the signature curve — a flat line, a staircase, a smooth cosine glide, sawtooth restarts, or the 1-cycle hump. On the right, that same curve drives a real gradient-descent run on an ill-conditioned bowl (steep in one direction, shallow in the other). A too-high constant rate oscillates and diverges up the steep wall; a too-low one crawls and never arrives; the scheduled rate — warm up so the first steps do not overshoot, then decay to near zero so it settles — lands cleanly in the basin. The value matters; the shape matters just as much.

🔨 Build it yourself

    
A schedule is nothing more than a rule for setting optimizer.param_groups[i]["lr"] before each step. You saw the shapes; here is how PyTorch gives them to you — first the from-scratch version so the mechanism is unmistakable, then the built-in schedulers, then how to chain a warmup in front of a cosine (the transformer recipe), OneCycleLR with its inverse momentum, and finally where scheduler.step() actually goes in the loop.

    
    
      
  1. 
        
    1
    From scratch: a schedule is just a function of the step
    
        
    Warmup is a linear ramp 0→base over the first W steps; after that, cosine annealing rides base→0. Compute the multiplier, write it into the optimizer. That is the whole idea — every scheduler below is a packaged version of this.
    
        
    import math

def lr_at(step, base, warmup, total):
    if step < warmup:                       # linear warmup: 0 -> base
        return base * step / max(1, warmup)
    progress = (step - warmup) / max(1, total - warmup)   # 0 -> 1
    return base * 0.5 * (1 + math.cos(math.pi * progress)) # cosine: base -> 0

base, W, T = 3e-4, 500, 10000
for step in [0, 250, 500, 5000, 10000]:
    for g in optimizer.param_groups:
        g["lr"] = lr_at(step, base, W, T)
    # ... optimizer.step() ...

    Built-in staircases: StepLR & ExponentialLR
    
        
    The classic decays. StepLR multiplies the LR by gamma every step_size epochs (a staircase); ExponentialLR multiplies by gamma every epoch (a smooth version of the same). Call scheduler.step() once per epoch here.
    
        
    import torch
opt = torch.optim.SGD(model.parameters(), lr=0.1, momentum=0.9)

# drop LR x0.1 at epoch 30 and 60:
sched = torch.optim.lr_scheduler.StepLR(opt, step_size=30, gamma=0.1)
# or a milestone list:
# sched = torch.optim.lr_scheduler.MultiStepLR(opt, milestones=[30,60,90], gamma=0.1)
# or smooth exponential decay every epoch:
# sched = torch.optim.lr_scheduler.ExponentialLR(opt, gamma=0.95)

for epoch in range(90):
    train_one_epoch(...)
    sched.step()                 # after the epoch
    print(epoch, sched.get_last_lr()[0])

    Cosine annealing & SGDR warm restarts
    
        
    CosineAnnealingLR glides base→eta_min over T_max steps — the modern default. CosineAnnealingWarmRestarts is SGDR: it anneals over a cycle of T_0 steps, then jumps the LR back up and starts again (T_mult lengthens each cycle). The jumps help the optimizer hop out of one basin into another.
    
        
    import torch
opt = torch.optim.SGD(model.parameters(), lr=0.1)

# smooth cosine to ~0 over the whole run:
cos = torch.optim.lr_scheduler.CosineAnnealingLR(opt, T_max=1000, eta_min=0.0)

# SGDR: restart every T_0 steps, each cycle 2x longer:
sgdr = torch.optim.lr_scheduler.CosineAnnealingWarmRestarts(
            opt, T_0=200, T_mult=2, eta_min=1e-5)

for step in range(1000):
    train_step(...)
    sgdr.step()                  # per-step scheduler

    Warmup + cosine — the transformer recipe
    
        
    This is what actually trains modern models. Either chain a LinearLR warmup in front of a CosineAnnealingLR with SequentialLR, or use the one call from HuggingFace transformers. Both give the ramp-then-glide shape.
    
        
    import torch
opt = torch.optim.AdamW(model.parameters(), lr=3e-4, weight_decay=0.1)
W, T = 500, 10000

warm = torch.optim.lr_scheduler.LinearLR(opt, start_factor=1e-8, total_iters=W)
cos  = torch.optim.lr_scheduler.CosineAnnealingLR(opt, T_max=T - W)
sched = torch.optim.lr_scheduler.SequentialLR(opt, [warm, cos], milestones=[W])

# --- or the one-liner every HF trainer uses: ---
from transformers import get_cosine_schedule_with_warmup
sched = get_cosine_schedule_with_warmup(
            opt, num_warmup_steps=W, num_training_steps=T)

    1-cycle — up then down, momentum inverse
    
        
    OneCycleLR ramps the LR from max_lr/div_factor up to max_lr, then anneals it down to near zero — and moves momentum the opposite way (high while LR is low, low while LR is high). Set max_lr from an LR range test; it drives the "super-convergence" fast-training regime.
    
        
    import torch
opt = torch.optim.SGD(model.parameters(), lr=0.1, momentum=0.9)

steps_per_epoch, epochs = len(train_loader), 30
sched = torch.optim.lr_scheduler.OneCycleLR(
            opt, max_lr=0.5,
            steps_per_epoch=steps_per_epoch, epochs=epochs,
            pct_start=0.3,           # fraction spent ramping up
            div_factor=25,           # start lr = max_lr / 25
            base_momentum=0.85, max_momentum=0.95)  # cycled inversely

for epoch in range(epochs):
    for batch in train_loader:
        train_step(batch)
        sched.step()                 # per BATCH for OneCycle

    Wiring it into the loop correctly
    
        
    The one bug everyone hits: scheduler.step() goes after optimizer.step(), and per-step schedulers (cosine warmup, OneCycle, SGDR) step every batch, not every epoch. ReduceLROnPlateau is the exception — it reads a validation metric.
    
        
    for epoch in range(epochs):
    for batch in train_loader:
        opt.zero_grad()
        loss = compute_loss(model, batch)
        loss.backward()
        opt.step()          # 1. update weights
        sched.step()        # 2. THEN advance the LR (per-step schedulers)

    # metric-driven schedulers step once per epoch, on a number:
    # plateau = ReduceLROnPlateau(opt, mode="min", factor=0.5, patience=5)
    # plateau.step(val_loss)

    print(epoch, opt.param_groups[0]["lr"])

🎉 Day 32 of DeepLearningFromZero done.
A learning-rate schedule is a function lr(t) the optimizer reads every step. One fixed value is a compromise: high enough to make early progress means overshooting near the minimum; low enough to settle means crawling at the start. Schedules resolve the tension in time. Warmup ramps 0→base so the first, noisiest steps (large-batch, Adam's cold variance estimates, deep transformers) do not blow up. Step and exponential decay are the classic staircases; cosine annealing ½·base·(1+cos(πt/T)) is the smooth modern default, gliding to near zero so the model settles; SGDR adds periodic restarts to escape basins; 1-cycle goes up-then-down with inverse momentum for fast training. The schedule also couples to batch size (linear scaling rule) and optimizer (Adam needs warmup more than SGD). The recipe that trains essentially every transformer is simply linear warmup then cosine decay. 👉 Tomorrow, Day 33: Knowledge Distillation — training a small student to match a big teacher's soft predictions.
