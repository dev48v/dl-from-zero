🔦 Beam Search — keep the top-k sequences, not just the top token

Greedy grabs the single best token every step. Beam keeps the best whole sequences.

A decoder (like Day 24's) gives a probability for every next token. Instead of a real net, this runs a tiny scripted language model: at each node there's a fixed distribution over a few words. Decoding is a search over the tree of sequences. This does real beam search — at every depth it expands each surviving hypothesis by all next tokens, scores each candidate by its cumulative log-probability (the sum of log p along the path), keeps the top-k and prunes the rest (greyed out). Set k = 1 and you get plain greedy decoding — watch it walk into a locally-best first word that misses the globally-best sentence a wider beam finds. Toggle length normalization to see the short-sequence bias fixed. All log-prob math and top-k pruning are genuine.

🔨 Build it yourself

    
You just watched real beam search prune a tree by cumulative log-prob. Here it is in Python: a greedy baseline, the beam core (expand, score, prune), an EOS pool, length normalization, and the sampling alternative. Drop in any model that returns next-token log-probs.

    
    
      
  1. 
        
    1
    The model interface: next-token log-probs
    
        
    Every decoder exposes the same thing: given the tokens produced so far, return a distribution over the next token. We work in log space from the start (log_softmax) so scoring a sequence is a sum, never a product that underflows to zero.
    
        
    import torch, torch.nn.functional as F

@torch.no_grad()
def next_logprobs(model, tokens, state):
    # tokens generated so far (begins with SOS)
    logits, state = model.step(tokens[-1], state)   # (vocab,)
    return F.log_softmax(logits, dim=-1), state      # log p over vocab

    A hypothesis, and the greedy baseline
    
        
    A hypothesis is a partial sequence plus its running score (sum of log p). Greedy decoding keeps exactly one hypothesis and always takes the argmax — fast, but it can never recover from a bad early commit.
    
        
    from dataclasses import dataclass

@dataclass
class Hyp:
    tokens: list          # e.g. SOS, 4, 1
    score: float = 0.0    # cumulative log-prob
    state: object = None
    done: bool = False

def greedy(model, max_len=20):
    h = Hyp(tokens=[SOS])
    for _ in range(max_len):
        lp, h.state = next_logprobs(model, h.tokens, h.state)
        tok = int(lp.argmax())            # take the single best token
        h.tokens.append(tok); h.score += float(lp[tok])
        if tok == EOS: break
    return h

    Beam search core: expand, score, prune
    
        
    Keep k hypotheses. Each step: expand every live hyp by every token, score each candidate (parent.score + log p), sort them all, keep the best k, discard the rest. Completed hyps (those that emitted EOS) move to a separate pool and stop expanding.
    
        
    def beam_search(model, k=4, max_len=20):
    beam = [Hyp(tokens=[SOS])]
    completed = []
    for _ in range(max_len):
        if not beam: break
        cands = []
        for h in beam:                                  # EXPAND
            lp, state = next_logprobs(model, h.tokens, h.state)
            for tok in range(VOCAB):                     # SCORE
                cands.append(Hyp(tokens=h.tokens + [tok],
                                 score=h.score + float(lp[tok]),
                                 state=state,
                                 done=(tok == EOS)))
        cands.sort(key=lambda c: c.score, reverse=True)  # PRUNE
        beam = []
        for c in cands[:k]:                              # keep top-k
            (completed if c.done else beam).append(c)
    return completed

    Length normalization: fix the short-sequence bias
    
        
    Because every extra token adds a negative log p, raw scores punish long sentences and beam search loves emitting EOS early. Divide the final score by length^alpha (Wu et al. use alpha near 0.6 to 0.7) before picking the winner — exactly the toggle in the LOOK demo.
    
        
    def best(completed, alpha=0.7):
    def norm_score(h):
        length = len(h.tokens) - 1          # exclude the SOS token
        return h.score / (length ** alpha)  # alpha=0 -> raw log-prob
    return max(completed, key=norm_score)

result = best(beam_search(model, k=4), alpha=0.7)

    The other family: sampling
    
        
    Beam optimizes for the likeliest text, which for open-ended generation is often bland and repetitive. Sampling instead draws from the distribution — temperature sharpens or flattens it; top-k / top-p (nucleus) trim the tail. This is what chat LLMs use for variety.
    
        
    def sample(model, temp=0.8, top_p=0.9, max_len=100):
    h = Hyp(tokens=[SOS])
    for _ in range(max_len):
        lp, h.state = next_logprobs(model, h.tokens, h.state)
        probs = F.softmax(lp / temp, dim=-1)          # temperature
        # nucleus (top-p): keep smallest set with cum prob above top_p
        sp, idx = probs.sort(descending=True)
        keep = (sp.cumsum(0) - sp) < top_p
        probs = torch.zeros_like(probs).scatter_(0, idx[keep], sp[keep])
        tok = int(torch.multinomial(probs / probs.sum(), 1))
        h.tokens.append(tok)
        if tok == EOS: break
    return h

    Pick the right decoder for the task
    
        
    Use beam when there is a mostly-correct answer to find: translation, summarization, speech-to-text, code. Use sampling when you want diversity and creativity: chat, stories, brainstorming. Many systems combine them — sample with a small beam, or beam-decode then rerank.
    
        
    # rule of thumb
translate(sentence)   -> beam_search(k=5)      # find the likeliest
summarize(document)   -> beam_search(k=4)
chat(prompt)          -> sample(temp=0.8, top_p=0.9)   # be interesting
brainstorm(prompt)    -> sample(temp=1.1, top_p=0.95)
